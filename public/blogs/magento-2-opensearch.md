# How I Solved Invalid Configurable Product Filter Combinations in Magento 2 OpenSearch

While working on a Magento 2 project recently, I came across an interesting search requirement that Magento's default layered navigation couldn't satisfy out of the box.

The project involved configurable products representing industrial fasteners. Customers frequently filtered products using attributes such as Length and Thread Size. The requirement sounded straightforward at first, but implementing it required a custom OpenSearch indexing and querying strategy.

## The Problem

Consider the following configurable product:

| Variant   | Length | Thread Size |
|-----------|--------|-------------|
| Variant A | 10     | M5          |
| Variant B | 20     | M6          |
| Variant C | 30     | M8          |

These are the only valid combinations available for purchase.

From a customer's perspective, the following combinations exist:

| Length | Thread Size | Valid |
|--------|-------------|-------|
| 10     | M5          | ✅     |
| 20     | M6          | ✅     |
| 30     | M8          | ✅     |
| 10     | M6          | ❌     |
| 20     | M5          | ❌     |
| 30     | M6          | ❌     |

Now imagine a customer applies these filters:

- Length = 10
- Thread Size = M6

The expected result is obvious:

> No products should be returned because no variant exists with Length 10 and Thread Size M6.

However, Magento still returns the configurable product.

## Why Magento Returns the Product

Magento indexes configurable product data in a way that allows layered navigation filters to operate efficiently.

For the configurable product above, Magento essentially knows:

**Lengths**
- 10
- 20
- 30

**Thread Sizes**
- M5
- M6
- M8

When a customer selects:
- Length = 10
- Thread Size = M6

OpenSearch evaluates these filters independently.

The configurable product contains:
- a child product with Length 10
- a child product with Thread Size M6

Because both conditions are technically satisfied somewhere among the associated simple products, the configurable product matches the query and appears in the search results.

This behavior is completely valid from Magento's perspective.

In fact, it is intentional. Magento's search index prioritizes filtering performance and faceted navigation over validating relationships between variant attributes.

The challenge was that the client wanted filtering based on actual variant combinations, not independent attribute matches.

## Designing the Solution

Instead of indexing Length and Thread Size separately for this use case, I decided to index the actual combinations available on each configurable product.

For the example above, the indexed values would become:
- `10_M5`
- `20_M6`
- `30_M8`

Now OpenSearch can determine whether a specific combination truly exists.

If a customer selects:
- Length = 10
- Thread Size = M6

The system will search for:
`10_M6`

Since that combination does not exist in the index, the configurable product will not be returned.

## Step 1: Extend the Product Search Index

I created a plugin on Magento's catalog search indexer:

```xml
<type name="Magento\CatalogSearch\Model\Indexer\Fulltext\Action\DataProvider">
    <plugin
        name="length_size_data_provider"
        type="Vendor\Module\Plugin\LengthSizeDataProvider"
    />
</type>
```

The plugin runs during product indexing.

For each configurable product, it loads all associated simple products and generates a list of valid attribute combinations.

The core logic looks like this:

```php
$combinations = [];

foreach ($associatedProducts as $simpleProduct) {
    $length = $simpleProduct->getData('length');
    $size = $simpleProduct->getData('thread_size');

    if ($length && $size) {
        $combinations[] = $length . '_' . $size;
    }
}

$result['length_size_combinations'] = implode(
    ', ',
    array_unique($combinations)
);
```

After indexing, a configurable product contains data similar to:

```json
{
    "length_size_combinations": [
        "10_M5",
        "20_M6",
        "30_M8"
    ]
}
```

At this point, OpenSearch knows the exact variant combinations available for the product.

## Step 2: Modify the OpenSearch Query

The next step was intercepting the OpenSearch query generated by Magento.

For this, I created a plugin on the OpenSearch mapper:

```xml
<type name="Magento\OpenSearch\SearchAdapter\Mapper">
    <plugin
        name="length_size_query"
        type="Vendor\Module\Plugin\LengthSizeQuery"
    />
</type>
```

The plugin inspects the currently applied layered navigation filters.

When both Length and Thread Size filters are present, it generates all possible selected combinations.

For example:

Length = 10,20  
Thread Size = M5,M6

Produces:
- `10_M5`
- `10_M6`
- `20_M5`
- `20_M6`

The plugin then adds a custom OpenSearch condition:

```php
$query['body']['query']['bool']['should'] = array_map(
    function ($combinationValue) {
        return [
            'terms' => [
                'length_size_combinations' => [$combinationValue]
            ]
        ];
    },
    $combinationValues
);

$query['body']['query']['bool']['minimum_should_match'] = 1;
```

This tells OpenSearch:
> "Return the product only if at least one selected combination exists in the indexed combination field."

## The Result

Let's revisit the original example.

Available combinations:
- `10_M5`
- `20_M6`
- `30_M8`

Customer selects:
- Length = 10
- Thread Size = M6

Generated combination:
- `10_M6`

OpenSearch checks the indexed combinations and finds no match.

Result:
**No products returned**

Exactly what the client expected.

## Scalability Considerations

One trade-off of this approach is increased index size because every valid attribute combination must be stored.

For our catalog, the number of variants per configurable product was relatively small, making the additional indexed data acceptable.

However, for configurable products with hundreds or thousands of variant combinations, it would be worth evaluating alternative indexing strategies to keep the search index efficient.

As with most search customizations, there is a balance between accuracy, index size, and query complexity.

## Benefits of This Approach

This implementation provided several advantages:

- No frontend customization was required.
- Existing layered navigation functionality remained unchanged.
- Filtering remained fully delegated to OpenSearch without additional PHP post-processing.
- Business rules were enforced directly at the search engine level.
- The solution scaled cleanly across the catalog.
- Product results accurately reflected purchasable variant combinations.

Another benefit is maintainability.

Because the combination field is generated during Magento's indexing process, changes to associated variants automatically become searchable after the normal reindex process runs.

## Final Thoughts

One thing I enjoy about Magento projects is that many customizations are not about fixing bugs. Instead, they involve adapting Magento's generic behavior to fit a specific business model.

In this case, Magento's default filtering logic was working exactly as designed. The client's requirement simply demanded a stricter interpretation of configurable product variants.

By indexing valid attribute combinations and incorporating them directly into the OpenSearch query, we were able to ensure that customers only saw products containing real, purchasable variant combinations.

The result was a more accurate search experience, cleaner product discovery, and behavior that aligned perfectly with the client's expectations.

Sometimes the best Magento customizations are not the ones that add new features—they are the ones that make existing features behave exactly the way the business needs.
